# 회의 개요 및 목적

- **배경:** 금융 및 공공 시장(전자정부 프레임워크, MVC 기반)에서의 생성형 AI 키워드 부상
- **목적:** xFrame 기반 UI 개발 환경에 생성형 AI를 도입하여 개발 생산성을 극대화하고, 개발자 편의성을 높이는 방안에 대한 아이디어 차원의 논의

# 회의내용

- xFrame 기반 UI 개발 환경에 생성형 AI 도입을 위한 요구사항 정리
    
    **1. UI 화면 생성 (Form Generation) - 비중 약 20%**
    
    - **핵심:** 기존 산출물(PPT, Excel, Figma 등)과 xFrame 스펙을 취합하여 **UI Form(XML)을 자동 생성**.
    - **기술:** S/W Base 접근 (XUL 등 XML 기반 UI 설계 툴 벤치마킹/참조).
    - **기대효과:** 수작업 코딩 없이 기획서/디자인에서 바로 화면 폼 정의.
    
    **2. API 및 개발 가이드 (RAG 기반) - 비중 약 20%**
    
    - **핵심:** 개발 중 필요한 정보(API 명세, 도움말)를 AI가 실시간 제공.
    - **방식:** RAG(검색 증강 생성) 기술 활용. xFrame 매뉴얼 및 DB를 연동한 **'xFrame GPT 챗봇'** 구축.
    - **기능:** 개발자가 질문하면 관련 API 가이드 및 예제 코드 제시.
    
    **3. 코딩 어시스턴스 (Coding Assistance) - 비중 약 30%**
    
    - **핵심:** 실제 비즈니스 로직 작성 지원.
    - **레퍼런스:** Cursor, Windsurf 등 최신 AI 코딩 도구의 기능 참조.
    - **방향:** 단순 자동 완성을 넘어 문맥을 이해하는 코딩 보조.
    
    **4. 테스트 케이스(TC) 작성 - 비중 약 20%**
    
    - **핵심:** 개발 완료 전후 후 테스트 시나리오 및 케이스 자동 생성.
- 제약 사항 및 리스크 (Risk & Constraints)
    - **도입 비용 (Cost):**
        - 금융/공공 특성상 클라우드(Claude, Public GPT 등) 사용 제약 존재.
        - On-premise 구축을 위한 **'AI Box(하드웨어)' 도입 비용**이 높음 → 전체 도입 비용 상승 요인.
    - **정확성 및 신뢰성 (Accuracy & Trust):**
        - "AI를 믿을 수 있는가?"
            - Data가 바뀌면 Output이 달라지는 환각(Hallucination) 및 비일관성 문제.
        - 속도(Speed)는 H/W 성능에 의존적이나, **정확성(Accuracy)**은 모델 성능 및 튜닝에 좌우됨.
        - 기존 RAG 만으로는 한계가 있을 수 있으며, 새로운 데이터/환경에 대한 적응력 검증 필요.
- SOFTBase 및 시선AI 입장 차이
    - SOFTBase
        - 기존 고객을 고려한 자사 제품(xFrame5)에 고객의 니즈(생성형 AI 활용)을 반영한 솔루션을 출시 또는 
        경쟁사 대비 기능 홍보에서 뒤쳐지기 않기 위해서 AI 활용 기능에 대한 프로토타입 개발이 필요한 상황
        - 기존 솔루션을 기반으로 기능을 확장하고, 영업할 수 있게 개발해야 하는데 LLM 에 대한 경험이 부족
        - 시선AI가 부족한 부분을 해결해서 자사 솔루션에 기능을 개선할 수 있도록 도움을 줄 수 있는가?에 대한 논의를 하고자 함
    - 시선AI
        - 시선AI 입장에서는 상기 툴(xFrame5)만을 위한 LLM 모델 개발 및 솔루션을 개발하기에는 부담이 큰 상황임
        - 따라서 좀 더 범용적인, 확장 가능한 형태의 솔루션(like `CusorAI` or `Claude Code` for SI Project)을 만드는 것이 낫다고 생각함
        - 또한 초기 도입 비용(인프라 구축 및 개발 비용)을 해당 솔루션에 어떻게 녹여 넣을지 사업적 논의(비용 회수 전략/구조)가 필요해 보임
- 참고자료(주용수M 작성)
    - 요구사항을 기반으로 한 시스템 아키텍처 (Conceptual Model)
        
        ![image.png](image.png)
        
        - **Client Side:** **xFrame Dev Studio** (UI 툴).
            - 개발자가 Tool 내에서 화면을 정의하고 UI를 구성.
        - **Server/AI Side:** **GPT 모듈 (AI Box)**.
            - **API Gateway:** 툴과의 통신 담당.
            - **Model:** 실제 추론을 담당하는 엔진.
            - **LLM + RAG:** 거대언어모델에 xFrame 전용 DB(매뉴얼, 소스 등)를 벡터화하여 결합.
    - 주용수M 개인의견
        - SI 사업 및 국내 고객사의 특성 상 AI 활용으로 기존보다 비용을 절감하고 좀 더 효율적인 기능을 제공하는 솔루션을 기대할 것으로 예상됨
        - 따라서 SI 사업을 위한 생성형 AI 코딩 어시스턴스 및 RAG 기반에 대한 수요는 존재하지만 구축 비용 또는 솔루션 비용에 대한 초기 비용이 문제가 될 가능성이 높다고 봄
            - 이를 해결하기 위해서는 비지니스 모델 및 사업 전략 수립이 필요함
        - 또한 시간이 지남에 따라서 생성형 AI를 위한 더 저렴하고 성능 좋은 H/W와 경량화된 LLM 모델은 개발될 것으로 보이나 이를 기다리고 있기만 해서는 안된다고 생각함
            - 일단 초기 시장 진입을 위해 빠른 프로토타이핑 개발이 필요함
                - 모델은 오픈소스 또는 ChatGPT/Claude/Gemini API를 기반으로 하고 자체 모델은 계속 R&D 진행
                - RAG나 코딩 어시스턴스 같은 기능은 직접 개발 - 오픈 소스 활용
        - 궁극적으로는 양사 간에 R&R 및 사업 방향에 대한 거시적인 관점에서의 논의가 필요하다고 사료됨
            - 목표
                - SI 사업을 위한 생성형 AI 코딩 어시스턴스 툴(IDE) 및 시스템(구축형 LLM? 포함) 개발
            - R&R
                - SOFTBase ← 코딩어시스턴스 IDE 개발(like CursorAI)
                - 시선AI ←  코딩어이스턴스 IDE를 위한 AI API 서비스 개발(RAG + LLM 등)
