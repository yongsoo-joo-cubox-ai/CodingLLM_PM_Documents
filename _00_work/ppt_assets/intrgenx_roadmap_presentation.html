왜 트랙 2가 필요한가

시장 패러다임 변화와 고객 반응의 Gap

IDE-First (현재 시장 주류)

Cursor, GitHub Copilot, Cline

  • 개발자가 주도, AI는 라인 단위 자동 완성
  • 인라인 Diff 제안, 시각적 직관성
  • 미시적 코드 수정에 최적화

Agent-First (차세대)

Claude Code, OpenCode → 시선AI 목표

  • 자연어로 목표 지시 → AI가 자율적으로 실행
  • 코드베이스 읽기, 계획 수립, 터미널 실행
  • 대규모 리팩토링, 거시적 작업에 최적
고객 반응의 Gap: IntraGenX 1.0은 설계 문서 → 코드 일괄 생성에 효과적이나, 고객들은 이미 Claude Code/Cursor급의 양방향 에이전틱 파트너에 익숙해져 있음. 단방향 엔진만으로는 시장 기대에 미달.

6대 USP는 유지하면서 에이전트 경험을 추가

트랙 1의 ①결정론적 출력 ②표준 강제 ③완전한 온프레미스 ④Spec-Driven ⑤감사 추적 ⑥LLM 추상화는 그대로 유지하고, 트랙 2에서 개발자가 체감하는 에이전트 경험을 추가합니다.

업스트림 추적 전략

  • git remote add upstream으로 원본 유지
  • Layer 1은 최소 수정 → rebase 충돌 최소화
  • 주요 릴리스 시 cherry-pick 방식 적용
  • 자동 CI에서 업스트림 변경 감지 알림

리스크 관리

  • 업스트림 breaking change → Layer 2/3에서 흡수
  • 코어 패치가 불가피할 경우 PR 제출 검토
  • MIT 라이선스 — 포크 및 상용화 법적 이슈 없음
  • 분기점 문서화: FORK_CHANGES.md 관리

LiteLLM 멀티모델 라우팅 심화

단일 엔드포인트 → 역할별 vLLM 클러스터 동적 분배

왜 프록시 레이어가 필요한가?

vLLM의 구조적 한계: 하나의 vLLM 인스턴스 = 하나의 베이스 모델. 이종 모델(7B + 20B + 32B)을 동시에 서빙하려면 별도의 라우팅 프록시가 필수.

config.yaml 실전 예시

model_list: # ─── 빠른 코드 작성/QA 검증용 소형 모델 ─── - model_name: seyeon-fast-coder litellm_params: model: openai/Qwen-2.5-Coder-7B api_base: "http://vllm-backend-small:8000/v1" # ─── 주력 코드 생성 모델 ─── - model_name: seyeon-code-gen litellm_params: model: openai/GPT-OSS-20B api_base: "http://vllm-backend-main:8001/v1" # ─── 복잡한 아키텍처 추론/Plan 에이전트용 ─── - model_name: seyeon-reasoning-agent litellm_params: model: openai/Seyeon-IntraGenX-32B api_base: "http://vllm-backend-large:8002/v1" # ─── 프로토콜 변환: claude-* → 로컬 sLLM 우회 ─── - model_name: claude-3-sonnet* litellm_params: model: openai/Seyeon-IntraGenX-32B api_base: "http://vllm-backend-large:8002/v1" router_settings: routing_strategy: "latency-based-routing" num_retries: 2 fallbacks: - seyeon-reasoning-agent: [seyeon-code-gen] - seyeon-code-gen: [seyeon-fast-coder]

4가지 핵심 기능

1. 모델 알리아싱
개발자는 localhost:4000/v1만 바라봄
백엔드 모델 버전 교체도 설정 파일 수정만으로 무중단
2. 프로토콜 변환
Anthropic Messages API → OpenAI 호환 포맷 자동 번역
claude-* 와일드카드로 에어갭 내 투명한 오프라인 운용
3. 부하 분산 + 폴백
latency-based / least-busy 전략
주력 모델 장애 시 자동 2차 모델 전환 → Zero Downtime
4. 거버넌스
Virtual Keys로 팀/프로젝트별 할당량 통제
RPM/TPM 기반 속도 제한 → 무한 루프 버그 방지

  • 비용 추적: 온프레미스에서도 GPU 시간/토큰 기반 내부 과금이 필요한가? (부서별 배분)

⚠️ MCP — 같은 프로토콜, 다른 용도

양쪽 모두 MCP(Model Context Protocol)라는 동일한 프로토콜 표준을 사용하되, 용도가 다릅니다:

트랙 1: MCP 서버가 코드 생성 엔진 역할 — Coco Engine이 xframe5-compiler, vue-compiler 등을 MCP 프로토콜로 호출
트랙 2: MCP 서버가 개발 도구 접근 역할 — 에이전트가 Jira, Confluence, Git 등 외부 서비스에 MCP 프로토콜로 접근

→ 동일한 프로토콜 위에서 서로 다른 도구(Tool)를 제공하는 것이 핵심 차이

시너지 포인트

  • 트랙 1에서 검증된 sLLM 파인튜닝 노하우 → 트랙 2 에이전트용 모델에 적용
  • UASL 스펙 → 트랙 2 에이전트의 자동 설계 분석에 활용 가능
  • 트랙 1의 6단계 검증 파이프라인 → 트랙 2의 보안 훅(3단계)에 재사용
  • 인프라 비용 절감: 동일 GPU 클러스터를 LiteLLM으로 양 트랙이 공유

레거시 모더니제이션 활용

Explore → Plan → Build 에이전트 분업으로 레거시 코드 전환

Step 1
🔍 Explore 에이전트 — 발견 및 추상화
레거시 코드베이스 전체를 순회하며 핵심 비즈니스 로직 추출 → 문서화. 구형 코드 속 숨겨진 의존성과 패턴을 자동 식별.
Step 2
📋 Plan 에이전트 — 구조 설계 및 전환 계획
추출된 로직 기반으로 최신 프레임워크(Spring Boot 3.x 등)에 맞는 디렉토리 구조, 모듈 분리 계획 수립. SI 하향식 업무 흐름에 완벽 부합.
Step 3
🔨 Build 에이전트 — 코드 생성 및 자가 검증
새 언어로 코드 작성 + 단위 테스트 자동 생성/실행. 전환 과정의 위험성을 극적으로 감소.
