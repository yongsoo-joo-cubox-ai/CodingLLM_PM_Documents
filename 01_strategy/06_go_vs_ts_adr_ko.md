## 2. 맥락

`05_track2_tech_strategy_ko.md`의 기술 전략 리서치는 TypeScript/OpenCode 포크를 유력한 구현 경로로 권고하였다. 그러나 실제 구현은 Go 기반 독자 구현(SecernCode)으로 진행되었으며 MVP 단계가 완료되었다.

의사결정 시점의 주요 외부 환경 변화:

- **OpenCode의 언어 전환**: OpenCode 자체가 2025년 5월~11월에 걸쳐 Go 기반 TUI에서 TypeScript/Bun 기반 아키텍처로 전면 전환하였다.
- **vLLM 단일 프로바이더 환경**: Coco의 폐쇄망 배포 환경에서는 다수의 AI 프로바이더를 지원하는 광범위한 SDK의 필요성이 낮다.
- **금융·공공 규제 요건 강화**: 설치 패키지 내 의존성 목록 제출 및 변경 불가능성 입증이 공급 계약의 필수 요건으로 부상하였다.

## 3. 고려된 대안

### 대안 A: OpenCode TypeScript 포크 (기각)

**장점**
- AI SDK를 통한 75개 이상의 LLM 프로바이더 지원
- 활발한 오픈소스 커뮤니티와 플러그인 생태계
- TypeScript 개발자 채용 풀 규모

**단점**
- Node.js/Bun 런타임 의존 — 폐쇄망 환경에서 런타임 별도 설치 및 관리 필요
- npm 의존성 트리 전체를 내부 미러에 구성해야 하는 운영 부담
- 보안 감사 범위 확대: 수백 개의 npm 패키지 각각에 대한 취약점 검토 필요

**기각 사유**: 폐쇄망 환경에서 npm 의존성 관리가 비현실적이며, 운영 복잡도가 인력 규모에 비해 과도하게 증가한다.

---

### 대안 B: SecernCode Go 독자 구현 (채택)

**장점**
- 단일 바이너리 배포 — 런타임 없이 바이너리 하나로 설치 완료
- zero-dependency 실행 환경 — 공급망 공격 표면 최소화
- 메모리 효율 및 낮은 레이턴시 — 느린 sLLM 환경에서 UI 반응성 유지
- 보안 감사 용이 — 정적 바이너리는 변경 불가능성 입증에 유리

**단점**
- AI SDK 부재 — 멀티 프로바이더 지원 구현 시 직접 코딩 필요
- Go 개발자 채용 풀이 Java·TypeScript 대비 협소함
- Bubble Tea TUI 라이브러리의 복잡 UI 확장 시 성능 한계 우려

**채택 사유**: 폐쇄망·온프레미스·금융/공공 규제 환경의 요구사항(단일 바이너리 배포, zero-dependency, 보안 감사 용이성)에 가장 직접적으로 부합한다.

## 4. 심층 검증 결과 (2026-03-26)

### 4.1 AI SDK 부재 — 단점 아님

SecernCode의 배포 환경은 vLLM 단일 프로바이더다. OpenAI 호환 API를 구현한 openai-go 라이브러리로 tool calling, 스트리밍, 재시도 로직을 완전히 구현하였으며 총 코드 규모는 1,494행이다.

AI SDK 도입 시 오히려 다음 문제가 발생한다.

- 미사용 프로바이더 코드가 바이너리에 번들링됨
- SDK 자체의 npm 의존성이 폐쇄망 미러링 부담으로 전환됨
- 버전 관리 복잡도 증가

### 4.2 플러그인 제약 — 오히려 보안 장점

SecernCode는 동적 플러그인 대신 세 가지 확장 메커니즘을 제공한다: MCP(Model Context Protocol) 서버, Custom Commands, Skills. 이 세 가지는 정적으로 정의되며 런타임에 외부 코드를 로딩하지 않는다.

동적 플러그인 부재는 공급망 공격 표면을 직접적으로 축소하며, 이는 금융권 고객의 보안 심사에서 명시적 강점으로 작용한다.

### 4.3 Bubble Tea 성능 — 현재 환경에서 위험 낮음

OpenCode의 Go TUI 성능 문제는 복잡한 멀티패널 UI와 고속 클라우드 LLM(응답 지연 낮음)의 조합에서 발생하였다. SecernCode의 환경은 단순 UI 구성과 느린 sLLM(응답 지연 높음) 조합이므로 동일 조건이 성립하지 않는다.

완화 경로도 존재한다.

- Bubble Tea v2 업그레이드 옵션 (API 호환성 대부분 유지)
- REST+SSE 기반 WebUI가 대체 인터페이스로 이미 구현되어 있어 TUI 성능 한계 도달 시 전환 가능

### 4.4 Go 개발자 풀 — 중간 리스크, 완화 가능

한국 채용시장에서 Go 대 Java의 공고 비율은 약 1:60 수준이다. 이는 실질적인 채용 리스크다.

완화 방안은 아래와 같다.

- Java 또는 TypeScript 3년 이상 경력자 채용 후 Go 전환 교육 2~3개월 적용 (당근마켓 선례 존재)
- SecernCode 핵심 코드 경로가 약 2,000행으로 온보딩 부담이 낮음
- Bus factor 해소를 위한 Go 개발자 추가 투입 계획 수립 필요

## 5. OpenCode Go→TS 전환과의 비교

### OpenCode 전환 타임라인

| 시점 | 사건 |
|------|------|
| 2025년 3월 | 순수 Go로 시작 (원래 이름: termai) |
| 2025년 5월 | AI SDK 도입을 위해 TypeScript/Bun으로 코어 엔진 전환 |
| 2025년 11월 | Go TUI(Bubble Tea) 완전 삭제 ("DELETE GO BUBBLETEA CRAP HOORAY") |

### OpenCode 전환 이유와 SecernCode 맥락 대비

| OpenCode 전환 이유 | SecernCode 해당 여부 | 판단 |
|-------------------|----------------------|------|
| AI SDK (75+ 프로바이더 지원) | 해당 없음 — vLLM 1개만 사용 | 위험 없음 |
| 플러그인 확장성 | 해당 없음 — MCP로 대체, 보안상 유리 | 위험 없음 |
| Bubble Tea 성능 한계 | 조건 불일치 — 단순 UI + 느린 sLLM | 위험 낮음 |
| 클라이언트/서버 분리 | 이미 구현 — REST+SSE WebUI 존재 | 위험 없음 |

결론: OpenCode가 Go를 버린 4가지 이유 모두 SecernCode 환경에서는 해당되지 않거나 위험이 낮다.

## 6. 결론 및 후속 조치

### 결론

Go 선택은 폐쇄망·온프레미스·규제 산업이라는 SecernCode의 특수 맥락에서 합리적인 결정이다. OpenCode의 Go 탈피는 범용 클라우드 SaaS 맥락에서 유효한 판단이나, SecernCode 환경에서는 동일한 동인이 작동하지 않는다.

TypeScript 전환을 재검토해야 하는 트리거 조건은 다음과 같다.

- 지원 LLM 프로바이더가 3개 이상으로 확장되는 경우
- Bubble Tea 성능 한계가 실제 고객 불만으로 발현되는 경우
- Go 개발자 채용 실패로 Bus factor 1 상태가 6개월 이상 지속되는 경우

### 후속 조치

1. Go 개발자 추가 투입 계획 수립 (Bus factor 해소)
2. Bubble Tea v2 업그레이드 로드맵 포함 여부 검토
3. 신규 개발자 온보딩 가이드 작성 (핵심 코드 경로 기준 ~2,000행)
